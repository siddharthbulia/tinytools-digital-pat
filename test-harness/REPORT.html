How these tests run. A Swift harness links the exact same supabase-swift 2.47.1 the shipped app uses, and spins up multiple independent anonymous identities ("devices") in one process via an in-memory auth store (the macOS Keychain blocks running two GUI copies). Each device runs a verbatim copy of the app's FriendStore logic — the same per-friendship presence channel topic, the same not-me matching, the same friendships postgres-changes watcher. Everything runs against the live Supabase project (cpfbipbdsokoshzgqvsr): real Realtime sockets, real RLS, real RPCs. A handful of states the public client cannot reach (an expired invite, deleting a user) use a service-role admin client. The 13-dimension scenario matrix behind this suite was generated by a multi-agent workflow (311 designed cases) and mined for coverage gaps.

Methodology & honest caveats

What is proven end-to-end (real SDK, live backend, multiple devices): the entire friend-graph data/realtime layer — invites, mutual friendship, live add/remove on both sides, RLS isolation (a non-party device receives zero events), presence-driven mood/character/cursor sync in both directions, offline/stale handling, multi-node topologies, reconnect/wake, the heartbeat backstop, and convergence under concurrent ops.

Local-UI bugs (#2 paste, #5 onboarding): verified by compiling the real Characters.swift and asserting the notification contract, plus asserting the Edit-menu paste mechanism (the standard paste(_:) selector that a focused text field handles — the exact thing that was missing). These are deterministic local code paths; the harness does not drive the actual NSWindow/AppKit event loop.

Note on the test environment: the project's anonymous-signup rate limit was temporarily raised from 30→2000/hr to allow ~70 fresh identities in one run, then restored to 30. No production data is touched — every identity is a throwaway anonymous user.
